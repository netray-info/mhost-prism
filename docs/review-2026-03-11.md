## Phase 1 — Critical & High: Security

These are the highest-risk findings. Workers A and B are independent; C can run at any time.

### 1.1 Update quinn-proto past RUSTSEC-2026-0037

**Worker**: A — Backend security
**Severity**: CRITICAL
**Files**: `Cargo.toml`, `Cargo.lock`

`quinn-proto` in the lock file is affected by RUSTSEC-2026-0037 (a remotely-triggerable
panic / DoS). Update the lock file to pull in a patched version. If `quinn-proto` is a
transitive dependency via `mhost`, add a `[patch.crates-io]` override or open an upstream
issue and pin to the minimum safe version locally.

Run `cargo audit` after updating to confirm no remaining advisories.

### 1.2 Prevent href injection via ifconfigUrl

**Worker**: B — Frontend security
**Severity**: HIGH
**Files**: `frontend/src/App.tsx` (or whichever component renders clickable IPs)

The `ifconfig_url` value from the server config is interpolated directly into `<a href>` on
IP addresses rendered in results. If the backend config is ever mis-set or the endpoint is
hijacked, `javascript:` or `data:` URIs can execute in the user's browser.

Sanitise: only allow `http://` or `https://` schemes before constructing the URL. Reject
or omit the link when the scheme does not match.

### 1.3 Run the Docker container as a non-root user

**Worker**: A — Backend security
**Severity**: HIGH
**Files**: `Dockerfile`

The final stage has no `USER` instruction; the process runs as root. Add a dedicated
system user in the image build and switch to it before `ENTRYPOINT`:

```dockerfile
RUN adduser --system --no-create-home --group prism
USER prism
```

This limits the blast radius of any path traversal or RCE.

### 1.4 Do not leak internal panic details through SSE streams

**Worker**: A — Backend security
**Severity**: HIGH
**Files**: `src/api/query.rs`, `src/api/check.rs`, `src/api/trace.rs`, `src/api/compare.rs`, `src/api/authcompare.rs`

If a spawned SSE task panics (e.g. due to an unexpected `unwrap()` in mhost internals),
the panic message can propagate into the `done` event's `warnings` field or be printed
to the SSE stream in raw form, exposing internal implementation details.

Wrap task bodies in `std::panic::catch_unwind` or use `tokio::task::JoinHandle::is_panicked()`
after `await`. On catch, emit a generic `done` event with `warnings: ["internal error"]`
and log the real cause at `error` level server-side only.

### 1.5 Harden the result cache key against enumeration

**Worker**: A — Backend security
**Severity**: HIGH
**Files**: wherever the result cache key is constructed (likely `src/api/query.rs` or the cache wrapper)

The cache key is derived directly from query parameters. Without a HMAC or keyed hash,
an attacker can enumerate cached keys to confirm whether specific queries have been issued
by other users (timing oracle). Either:

- Accept this as a documented trade-off (DNS caches are inherently observable), or
- Key the cache with an opaque server-side secret so external enumeration is not possible.

Document the chosen posture in a comment at the key-construction site.

### 1.6 Remove dead hot-reload rate-limiter or wire it correctly

**Worker**: A — Backend security
**Severity**: HIGH
**Files**: `src/security/rate_limit.rs`, `src/main.rs`

A hot-config-reload path exists in AppState (an `ArcSwap<Config>`) but the rate-limiter
layers are built once at startup and never updated when the config changes. The hot-reload
code is therefore dead — it updates state that nothing reads. Either:

- Remove the hot-reload infrastructure entirely (YAGNI), or
- Rebuild the governor layers on each reload and swap them atomically.

Dead code that looks live is a maintenance hazard.

## Phase 2 — High: Infrastructure, Observability, and UX

Workers D, E, and F are independent and can merge in any order.

### 2.1 Remove or wire the unwired AppState infrastructure

**Worker**: D — Dead infrastructure
**Severity**: HIGH
**Files**: `src/main.rs`, `src/api/query.rs`, `src/api/check.rs`, `src/api/trace.rs`, `src/api/compare.rs`, `src/api/authcompare.rs`

`AppState` contains `resolver_pool`, `query_dedup`, and `hot_state` fields that are
never accessed by any request handler. They consume memory, introduce lock contention
risk (the `ResolverPool` uses a write-biased lock even on reads — see 6.2), and imply
capabilities the service does not actually have.

Decision required: either wire them into the handlers (requires careful design and tests
to avoid regressions) or remove them and the associated files until they are genuinely
needed (YAGNI). A hybrid — keep `result_cache`, remove the rest — may be the right call.

### 2.2 Fix double enrichment fetch in check.rs

**Worker**: D — Dead infrastructure
**Severity**: HIGH
**Files**: `src/api/check.rs`

The check endpoint calls the enrichment helper twice for the same set of IPs — once at the
end of the DNS sweep and once after the DMARC lint step. Both calls are awaited serially,
doubling latency (2 × `enrichment_timeout_ms`) for no benefit. Collect all IPs after all
DNS steps complete, then call enrichment exactly once before emitting the `done` event.

### 2.3 Fix Prometheus label cardinality explosion

**Worker**: D — Dead infrastructure
**Severity**: HIGH
**Files**: `src/main.rs` (metrics middleware), wherever `prism_http_requests_total` is emitted

The `path` label on `prism_http_requests_total` uses the raw request path. Any endpoint
that embeds user-controlled data (e.g. `GET /api/query?q=<domain>`) will create one
Prometheus label series per unique query string, exhausting cardinality limits.

Normalise the `path` label to the route template, not the raw URI. Use axum's
`MatchedPath` extractor (available in Tower middleware) and fall back to `"unknown"`.

### 2.4 Fix /api/ready returning 503 during circuit-breaker HalfOpen

**Worker**: E — Observability
**Severity**: HIGH
**Files**: `src/api/meta.rs`

The readiness probe returns HTTP 503 when any circuit breaker is in HalfOpen state.
HalfOpen means the breaker is actively probing — the service is recovering and should
receive traffic. Kubernetes will remove the pod from the load-balancer endpoint list
while it is trying to heal itself.

Return 200 for HalfOpen. Only return 503 for Open state (confirmed failing).

### 2.5 Log SSE task errors server-side

**Worker**: E — Observability
**Severity**: HIGH
**Files**: `src/api/query.rs`, `src/api/check.rs`, `src/api/trace.rs`, `src/api/compare.rs`, `src/api/authcompare.rs`

When an SSE background task fails (resolver error, timeout, circuit-breaker open), the
error is emitted to the client as a structured event but never logged on the server. An
operator seeing elevated error rates in metrics has no log trail to investigate the cause.

Add `tracing::warn!` (resolver errors, timeouts) or `tracing::error!` (unexpected failures)
at the point where each task result is inspected before sending the `done` event.

### 2.6 Add per-endpoint error counters

**Worker**: E — Observability
**Severity**: HIGH
**Files**: `src/api/check.rs`, `src/api/trace.rs`, `src/api/compare.rs`, `src/api/authcompare.rs`

`prism_queries_total` is only emitted for `/api/query`. The check, trace, compare, and
authcompare endpoints have no dedicated counters, making it impossible to tell which
endpoint is generating errors or load from Prometheus alone.

Add `prism_queries_total{endpoint="check|trace|compare|authcompare", status="ok|error"}`
at each endpoint's completion path.

### 2.7 Fix OTel shutdown not flushing spans

**Worker**: E — Observability
**Severity**: HIGH
**Files**: `src/telemetry.rs:102`

`shutdown()` calls `global::set_tracer_provider(NoopTracerProvider::new())` which drops
the old provider, but the OTLP HTTP exporter uses a background thread and `drop` is not
guaranteed to flush before the process exits. Replace with an explicit
`provider.shutdown()` call (which blocks until the batch queue drains) stored in a
`OnceCell<SdkTracerProvider>`.

### 2.8 Add ARIA roles and labels to interactive elements

**Worker**: F — UX
**Severity**: HIGH
**Files**: `frontend/src/components/ResultsTable.tsx`, `frontend/src/App.tsx`, `frontend/src/components/TraceView.tsx`

Several interactive elements lack accessible roles:

- Tab bar: add `role="tablist"`, `role="tab"`, `aria-selected`, and `aria-controls` to the
  mode-selector tabs.
- Expandable table rows: add `role="button"`, `tabIndex={0}`, `aria-expanded`, and
  keyboard handlers (`Enter` / `Space`) to record-group headers.
- HopCard in TraceView: add `role="button"`, `tabIndex={0}`, `aria-label` with hop number.
- Query input `<div contenteditable>` / CodeMirror wrapper: add `aria-label="DNS query"`.

### 2.9 Add role="alert" to error banner

**Worker**: F — UX
**Severity**: HIGH
**Files**: `frontend/src/App.tsx`

The error banner shown when a request fails has no ARIA live region. Screen readers will
not announce it when it appears. Add `role="alert"` (or `aria-live="assertive"`) to the
banner container.

### 2.10 Fix mobile layout for transport and auth comparison grids

**Worker**: F — UX
**Severity**: MEDIUM
**Files**: `frontend/src/components/TransportComparison.tsx`, `frontend/src/components/AuthComparison.tsx`, `frontend/src/styles/global.css`

The four-column transport grid and two-column auth grid have no responsive breakpoints.
On screens narrower than ~600 px the columns overflow horizontally and the data is
unreadable. Add a `@media (max-width: 640px)` rule that stacks columns vertically or
switches to a single-column accordion layout.

### 2.11 Disable submit button during in-flight requests

**Worker**: F — UX
**Severity**: MEDIUM
**Files**: `frontend/src/App.tsx`, `frontend/src/components/QueryInput.tsx`

The submit button and the `r` keyboard shortcut remain active while a stream is in
progress, allowing users to issue overlapping requests unintentionally. Pass
`disabled={status() === 'loading'}` to the button and guard the keyboard shortcut
handler on the same condition.

### 2.12 Trap focus in help modal

**Worker**: F — UX
**Severity**: MEDIUM
**Files**: `frontend/src/App.tsx`

The help modal does not trap focus. Keyboard users can tab behind the modal overlay to
the main content. Add a focus trap:

- On open, move focus to the first focusable element inside the modal.
- Intercept `Tab` / `Shift+Tab` to cycle within the modal.
- On close, return focus to the element that triggered the open.
- Add `role="dialog"`, `aria-modal="true"`, `aria-labelledby` pointing to the modal title.

## Phase 3 — Medium: Test Coverage and Engineering Cleanup

Workers G and H are independent.

### 3.1 Add SSE stream-level integration tests

**Worker**: G — Tests
**Severity**: HIGH
**Files**: `src/api/query.rs`, `src/api/check.rs`, `src/api/trace.rs`

There are no tests that actually consume an SSE stream and assert on event structure.
Use `axum_test::TestServer` with a mock `ResolverGroup` (or a local stub) to assert:

- A query produces at least one `batch` event followed by a `done` event.
- A policy violation produces a 422 response, not an open stream.
- A rate-limit hit produces a 429 with `Retry-After`.
- The `done` event JSON matches the `DoneEvent` schema (`total_queries`, `duration_ms`, etc.).

### 3.2 Add result cache unit tests

**Worker**: G — Tests
**Severity**: MEDIUM
**Files**: wherever the result cache lives

Test that:
- A cached entry is returned on the second identical query.
- A cache miss calls the resolver.
- A TTL-expired entry is evicted before returning.
- Concurrent requests for the same key do not issue duplicate upstream queries (dedup
  behaviour, if wired).

### 3.3 Add /api/ready 503 path test

**Worker**: G — Tests
**Severity**: MEDIUM
**Files**: `src/api/meta.rs`

The readiness endpoint has no test for the 503 path (open circuit breaker). Add a unit
test that puts a breaker into Open state and asserts `GET /api/ready` returns 503.
After fix 2.4 lands, also assert that HalfOpen returns 200.

### 3.4 Remove live network dependency from unit tests

**Worker**: G — Tests
**Severity**: MEDIUM
**Files**: wherever the offending test lives (identified by the testing lens agent)

At least one unit test issues a real DNS query over the network. This makes the test
suite non-deterministic, slow, and unusable in air-gapped CI environments. Replace with
a stub resolver or a pre-recorded response fixture.

### 3.5 Add CI coverage gate

**Worker**: G — Tests
**Severity**: MEDIUM
**Files**: `.github/workflows/ci.yml`

The CI pipeline has no minimum coverage threshold. Add `cargo-llvm-cov` or `tarpaulin`
to the `ci` job and fail the build when line coverage falls below a chosen floor (e.g.
60% to start; raise over time). Record the baseline first to avoid blocking the pipeline
with a target that is already missed.

### 3.6 Fix write-biased lock in ResolverPool

**Worker**: H — Engineering cleanup
**Severity**: HIGH
**Files**: wherever `ResolverPool` is implemented (likely `src/main.rs` or a dedicated module)

`ResolverPool::get()` acquires a write lock even when it will only read the pool and
return an existing entry. Under concurrent load, this serialises all pool lookups through
a single exclusive lock. Change read-path accesses to use `RwLock::read()` and only
upgrade to write on an actual insert.

### 3.7 Replace DefaultHasher in QueryDedup

**Worker**: H — Engineering cleanup
**Severity**: MEDIUM
**Files**: wherever `QueryDedup` is implemented

`std::collections::HashMap` / `DashMap` defaults to `DefaultHasher`, which uses a
randomly-seeded SipHash at process startup. This is intentional for DoS resistance in
user-facing maps, but for an internal dedup map keyed on structured query data it adds
unnecessary overhead. Switch to `AHashMap` (already a transitive dependency via `mhost`)
for measurable throughput improvement on hot paths.

### 3.8 Replace sync Mutex on async hot path

**Worker**: H — Engineering cleanup
**Severity**: MEDIUM
**Files**: AppState / wherever the sync `Mutex` wraps a structure accessed in async context

Using `std::sync::Mutex` in an async task is safe only if the lock is never held across
an `await` point. Audit every `Mutex` in `AppState` and handler code. Any lock held
across an `.await` must be replaced with `tokio::sync::Mutex` (or the data structure
redesigned to avoid the lock entirely).

### 3.9 Deduplicate private-IP validation logic

**Worker**: H — Engineering cleanup
**Severity**: MEDIUM
**Files**: `src/security/query_policy.rs`, `src/dns_trace.rs`

`is_allowed_target` in `query_policy.rs` and a similar inline check in `dns_trace.rs`
both enumerate RFC 1918 / loopback / link-local / ULA ranges. The logic diverged when
the ULA fix (review 2025-03-07 §1.3) was applied to `query_policy.rs` but not to the
trace inline check.

Consolidate: remove the inline copy in `dns_trace.rs` and call `is_allowed_target` for
all IP validation (as was done for glue addresses in §1.4 of the previous fix plan).

## Phase 4 — Low: Documentation

These are documentation-only changes; they carry no code risk and can be drafted at
any time, including in parallel with earlier phases.

### 4.1 Remove dead docs/sdd.md reference from CLAUDE.md

**Files**: `CLAUDE.md`

The CLAUDE.md still references `docs/sdd.md` in several places. The file was moved to
`docs/done/sdd-2025-03-07.md` and is no longer the live design document. Update all
references to point to `docs/done/sdd-2025-03-07.md` (historical) and clarify that
design authority now lives in the README + code comments.

### 4.2 Sync README config reference

**Files**: `README.md`

The full config reference in the README has diverged from the actual defaults in
`prism.example.toml` on at least two values. Do a pass comparing every commented default
in `prism.example.toml` against the README table and update mismatches.

### 4.3 Fix hardening checklist CIDR claim

**Files**: `README.md`

The hardening checklist says `trusted_proxies` supports CIDR ranges. The code only
accepts individual IP addresses (`IpAddr::from_str`, not a CIDR parser). Fix the
checklist to say "individual IP addresses" and cross-reference the limitation note added
in the 2026-03-11 code changes.

### 4.4 Add CONTRIBUTING.md, SECURITY.md, and CHANGELOG.md stubs

**Files**: `CONTRIBUTING.md` (new), `SECURITY.md` (new), `CHANGELOG.md` (new)

A public GitHub repo without these files fails basic open-source hygiene checks and
makes it harder for contributors to know how to submit bug reports or patches.

- `SECURITY.md`: responsible disclosure policy (email or issue template), scope (what is
  in/out), response SLA.
- `CONTRIBUTING.md`: build prerequisites, `make ci` as the pre-PR gate, coding
  conventions pointer to CLAUDE.md.
- `CHANGELOG.md`: start with `v0.1.0` entry summarising the completed phase work.
